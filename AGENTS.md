# AGENTS.md

Guidance for coding agents. Human docs: [README.md](README.md) (Norwegian) and
https://docs.digdir.no/docs/eFormidling/.

## What this is

**eFormidling Integrasjonspunkt** (Digdir): a Spring Boot 3.5 / Java 21 message
exchange point for the Norwegian public sector. Clients send/receive messages
through its REST API ("NextMove API"); the application routes them over transport
channels — DPO (Altinn), DPV, DPI (digital post), DPF/DPFIO (Fiks), DPE (eInnsyn),
DPH (helse) — chosen per receiver via the Service Registry lookup.

Runs on port **9093** locally. Embedded ActiveMQ for queuing, H2 by default
(Postgres/MariaDB/MySQL/MSSQL in production). Docs/commits are in Norwegian;
code and identifiers are in English.

## Codebase map

Multi-module Maven build (`no.difi.meldingsutveksling`, version `DEV-SNAPSHOT`):

| Module | Contents |
|---|---|
| `integrasjonspunkt/` | Main app. Entry: `IntegrasjonspunktApplication`. REST controllers in `nextmove/v2/` (`NextMoveMessageOutController`, `NextMoveMessageInController`, `CapabilitiesController`), `status/service/`, `webhooks/`. Config profiles in `src/main/resources/config/`, Liquibase changelogs in `src/main/resources/db/changelog/` |
| `api/` | Internal SPI interfaces: `ConversationStrategy` + per-channel strategies/pollers (`DpeConversationStrategy`, `DpfPolling`, …) |
| `nextmove/` | Message model and channel logic (`nextmove/`, `receipt/`, `arkivmelding/`) |
| `domain/` | JPA entities and shared domain types |
| `common/`, `common-spring/` | Utilities; `common-spring` holds `IntegrasjonspunktProperties` — the typed view of all `difi.move.*` config — and its validator |
| `dpi-client/`, `dph-client/`, `altinn-v3-client/`, `fiks-meldingsformidler/`, `post-til-private/` | Channel-specific transport clients |
| `service-registry-client/` | Lookup of receiver capabilities/certificates |
| `validation/`, `dokumentpakking/` | Payload validation; ASiC-E packaging/signing |
| `web/` | Admin UI (Thymeleaf). Runs standalone: `ThymeleafApplication` |

## Setup gotcha (read first)

Dependencies resolve from GitHub Packages (`maven.pkg.github.com/felleslosninger/*`),
which needs a `~/.m2/settings.xml` server entry with a GitHub token
(`read:packages`). A 401/403 during dependency resolution means missing/expired
token — stop and ask the user; there is no workaround.

Requires JDK 21 and Maven 3.9+.

## Build, run, test

```bash
mvn clean install                      # full build + unit tests
mvn verify                             # + integration tests; this is what CI runs — run before declaring done
                                       # (fast: ~1.5 min incremental on a dev laptop; no Docker required)
mvn test -pl integrasjonspunkt -Dtest=SomeTest          # one unit test class
mvn verify -pl integrasjonspunkt -Dit.test=SomeIT       # one integration test class

# Run locally (staging = test-environment profile; needs local config, see below)
java -Dspring.profiles.active=staging -jar integrasjonspunkt/target/integrasjonspunkt.jar

# Admin UI standalone with fake backend + hot reload (see web/README.md)
cd web && mvn spring-boot:run -Dspring-boot.run.profiles=reload   # http://localhost:8080

# REST API docs (Spring REST Docs — snippets are generated BY the tests, so tests must run)
mvn clean package -Prestdocs && open integrasjonspunkt/target/generated-docs/restdocs.html
```

Test taxonomy:

- `*Test.java` → unit tests, maven-surefire, run on `mvn test`.
- `*IT.java` → integration tests, maven-failsafe, run on `mvn verify`.
- **Cucumber** end-to-end features: `integrasjonspunkt/src/test/resources/cucumber/*.feature`
  (one per channel/direction, e.g. `nextmove_send_dpo.feature`), executed by
  `RunCucumberIT` with glue in `no.difi.meldingsutveksling.cucumber`. Scenarios
  tagged `@Ignore` are skipped. Run a subset with
  `mvn verify -pl integrasjonspunkt -Dit.test=RunCucumberIT -Dcucumber.filter.tags=@yourtag`.
  When changing channel behavior, extend the matching feature file.
- Database tests in `integrasjonspunkt/.../databases/*DatabaseIT.java` use
  Testcontainers (need Docker) and are `@Disabled` by design — leave them disabled
  in committed code. A green `mvn verify` reports ~12 skipped ITs in
  `integrasjonspunkt`; that is normal (`@Ignore` scenarios + disabled DB tests),
  not a sign of a broken run.
- Test output is extremely verbose (Hibernate SQL debug logging, thousands of
  lines). Redirect build output to a file and grep for `Tests run`, `FAILURE`,
  `ERROR`, and the reactor summary instead of reading it linearly.

Useful when running locally: `http://localhost:9093/` (status page),
`/conversations`, `/api/statuses`, `/manage/health`, `/manage/logfile`.

## Common tasks

- **Add/change a config property:** add the field to `IntegrasjonspunktProperties`
  (`common-spring`), default in
  `integrasjonspunkt/src/main/resources/config/application.properties`, and
  validation in `IntegrasjonspunktPropertiesValidator` if required. All app config
  lives under the `difi.move.*` prefix.
- **Database schema change:** schema is managed by Liquibase
  (`integrasjonspunkt/src/main/resources/db/changelog/`) *plus*
  `hibernate.ddl-auto=update`. Add a changelog for any entity change; never edit an
  already-released changeset. Must work on H2, Postgres, MariaDB, MySQL, and MSSQL.
- **Change a REST endpoint:** controllers in `integrasjonspunkt` under
  `nextmove/v2/` or `status/service/`. The REST Docs tests double as API
  documentation — update them together with the endpoint or `-Prestdocs` breaks.
- **Channel-specific behavior:** find the channel's `ConversationStrategy`/polling
  implementation via the interfaces in `api/`, implementation in `nextmove/` or the
  channel's client module.

## Local config — never commit

Local overrides go in `integrasjonspunkt-local.properties` (or `.yml`) in the repo
root, auto-imported by the `dev`/`staging`/`production` profiles. The root
typically contains untracked developer files (`application-*.properties`,
`integrasjonspunkt-local*`), often with org numbers and keystore passwords:

- Never commit these, never copy their values into code, tests, or commit messages.
- `integrasjonspunkt-local.sample.properties` documents the available settings.
- Certificates/keystores (`certs/`, `*.jks`, `*.p12`) must never be committed.

## Conventions

- Commit messages and PR titles start with the Jira key:
  `MOVE-5100 Fix popMessage delete rolled back when asic.zip is missing`. Ask for
  the key if you don't have one. PRs target `main`.
- Lombok everywhere (`@RequiredArgsConstructor` for injection, `@Slf4j`, `@Data`);
  `lombok.config` enables chained accessors and copies `@Qualifier`/`@Value`/`@Lazy`
  onto constructors. Don't hand-write boilerplate Lombok can generate.
- No checkstyle/formatter config is enforced — match the surrounding file exactly.
- Update `CHANGELOG.md` for user-visible changes.
- Releases are cut manually via GitHub Releases with `vX.Y.Z` tags and signed
  artifacts — agents must not attempt any part of the release process.
- Dependency-scan suppressions live in `supression.xml` (sic — keep the existing
  filename); never add a suppression without flagging it to the user.

## Domain vocabulary

Norwegian terms you'll meet in code and features: *melding* = message,
*kvittering* = receipt, *arkivmelding* = archive message format,
*innsynskrav* = freedom-of-information request, *meldingsutveksling* = message
exchange, *mottak* = receiving. Channel names (DPO, DPV, DPI, DPF, DPE, DPH) are
"digital post" variants — the channel matrix is at docs.digdir.no.
